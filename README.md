# Тестовое задание Mindbox (SRE)

Решение задачи по развертыванию веб-приложения в Kubernetes с учетом требований отказоустойчивости и оптимизации ресурсов.

## Принятые архитектурные решения

1. **Graceful Shutdown (`preStop` hook + `terminationGracePeriodSeconds`):** Обеспечивает нулевой downtime при автомасштабировании HPA и деплоях. Балансировщик успевает убрать IP пода из роутинга до его фактической остановки.
2. **Memory QoS (Guaranteed):** Requests и Limits по памяти равны (`128Mi`). Это защищает поды от случайного OOMKill при нехватке ресурсов на ноде, перенося их в класс Guaranteed (по памяти).
3. **Startup Probe:** Учитывает требование к инициализации (5-10 сек), давая приложению запас времени на старт без риска быть убитым `livenessProbe`.
4. **HPA Behavior:** Настроено быстрое масштабирование вверх (мгновенно) при пиках и плавное масштабирование вниз (окно 5 минут), чтобы HPA не "дергал" поды при кратковременных просадках трафика.
5. **Topology Spread Constraints:** Жесткое распределение (`maxSkew: 1`, `DoNotSchedule`) по 3 зонам для High Availability, и мягкое распределение по 5 нодам.

## Пути дальнейшей оптимизации (Future Work)
* **KEDA (Kubernetes Event-driven Autoscaling):** Учитывая четкий дневной цикл нагрузки, логично использовать KEDA (cron trigger) для пре-скейлинга подов за 15-30 минут до начала "дня", чтобы избежать деградации ответа у первых пользователей во время масштабирования.
* **In-place Pod Resource Updates (KEP-1287):** Для решения проблемы "тяжелых" первых запросов в будущем можно динамически снижать CPU Limits без рестарта пода, экономя ресурсы кластера.
